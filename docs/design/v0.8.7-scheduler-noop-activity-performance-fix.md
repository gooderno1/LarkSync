# LarkSync v0.8.7 启动调度与空活动性能修复

## 1. 问题结论

`v0.8.6` 已消除问题历史重放和 JSONL 回填对启动请求的阻塞，后端可在约 1–2 秒内完成健康检查；当前剩余的“启动仍慢”和“很多上 0 下 0”来自同步调度与活动口径。

正式版只读实测：

- 升级启动后，所有秒/小时周期任务在 12 秒保护期结束时立即执行，没有继承升级前的下次执行时间。
- 一个定时下载运行检查 2,471 个对象，耗时约 434.5 秒；仅下载 2 个，跳过 2,469 个。
- 一个定时上传运行处理 2,251 个路径，耗时约 320.8 秒；上传 16 个，跳过 2,233 个。
- 对上述 2,233 个“内容未变化”路径抽样 500 个，`sync_links.local_size/local_mtime` 与磁盘当前值 500/500 一致，本可在转换、资源检查和云端比对前过滤。
- `SyncTaskRunner._persist_run_finished()` 强制写入 `run_kind=activity`、`has_activity=true`，覆盖了运行服务对纯检测运行的正确推断。
- 删除墓碑尚未到宽限期或重试时间时，仍会让每分钟上传任务进入一次运行；同一墓碑被刷新时又会重复记录 `delete_pending`。
- 活动列表摘要只显示上传、下载和异常数；实际存在 `deleted/delete_pending/delete_failed` 的运行仍显示为“上 0 下 0”。

## 2. 修复目标

- 应用重启或自动升级后，小时级下载任务继续等待剩余周期，不立即执行全量扫描。
- 上传与下载分别保存调度时间，不因双向任务互相覆盖。
- 文件系统产生的无效触碰事件在本地签名层过滤，不进入 Markdown 转换或云端请求。
- 没有上传、下载、删除、冲突、失败或待删除事实的成功定时运行不进入活动列表。
- 同一待删除事实只在首次建立时记录一次；未到期的墓碑不触发同步运行。
- 活动摘要完整展示删除、待删除和删除失败，避免真实删除活动被误读为“上 0 下 0”。
- 兼容所有旧版本用户的历史数据，不写入任何特定用户、路径或任务规则。

## 3. 启动调度检查点

### 3.1 存储

使用现有 `sync_meta` 表保存每个任务、每个方向的最后调度尝试时间：

```text
sync_schedule_checkpoint_v1:{task_id}:upload
sync_schedule_checkpoint_v1:{task_id}:download
```

上传和下载使用独立键。配置改变时，以当前检查点和新周期重新计算剩余等待时间。

### 3.2 首次兼容

- 检查点存在：`remaining = last_attempt + interval - now`。
- 检查点不存在：优先读取相同任务和相同方向最近一次 `scheduled_upload/scheduled_download` 运行时间；旧库没有方向运行历史时再以任务 `last_run_at` 为兼容基线。
- 检查点不存在且任务从未运行：立即执行，保证新建任务仍能完成首次同步。
- 每次调度调用结束后，无论本次有无文件变化，都写入当前方向的检查点。
- 手动同步不改变周期检查点。

### 3.3 边界

- 日历型计划继续按下一日历时间计算，本次不改变。
- 进程运行期间仍保持当前“执行完再等待一个周期”的语义。
- 调度检查点只控制何时检查，不改变 Watchdog 对本地变化的收集。

## 4. 上传无变化快速过滤

周期上传从静默窗口取得 `ready_paths` 后，一次读取任务的 `sync_links`，对每个路径执行：

1. 路径不存在、不是普通文件或没有映射：保留，由原流程处理。
2. `local_size` 和 `local_mtime` 都存在，且大小相同、mtime 误差不超过 1 毫秒：从本轮队列移除。
3. 任一签名不一致：保留，继续原有内容哈希、资源签名、冲突保护和上传流程。

被快速过滤的路径属于“检测无变化”，不创建运行、不写文件事件。该规则不依赖扩展名，对 Markdown 和普通文件均适用；不能只按大小过滤。

## 5. 删除墓碑去重与到期判断

- `has_pending_tombstones(task_id)` 只查询 `expire_at <= now` 的 pending/failed 项。
- `create_or_refresh()` 返回 `created` 标记：
  - 首次创建 pending 墓碑：`created=true`。
  - 刷新同一 pending 墓碑：`created=false`。
- 三个删除发现入口仅在 `created=true` 时记录 `delete_pending` 事件。
- 刷新保留最早 `detected_at` 和最早 `expire_at`，宽限期不会被周期检查无限延长。
- 飞书删除返回 `not found. token=...` 时按幂等成功处理并清理墓碑，不再每 5 分钟重复记录删除失败。
- 到期执行、失败重试和严格删除策略保持不变。

## 6. 活动口径与历史兼容

### 6.1 新运行

运行结束时不再强制覆盖 `run_kind/has_activity`，由 `SyncRunService.finish_run()` 根据以下事实推断：

- `failed/cancelled/running` 状态；
- 非空 `last_error`；
- `uploaded/downloaded/deleted/conflict/delete_pending/delete_failed/failed` 任一计数大于 0。

其余成功运行记为 `legacy_check`、`has_activity=false`。

### 6.2 历史迁移

数据库 schema v4 对所有用户执行一次通用重分类：

- 仅处理成功、无错误、上述活动计数总和为 0 的运行；
- 将其改为 `legacy_check`、`has_activity=0`；
- 失败、中断、上传/下载/删除/冲突/待删除/删除失败运行保持可见；
- 不删除运行或事件，不修改问题事实。

## 7. 页面摘要

运行卡片按非零项展示：

```text
上 2 · 下 1 · 删 3 · 待删 1 · 删失败 1 · 异常 2
```

- 上传、下载继续固定显示，便于快速比较。
- 删除、待删除、删除失败为 0 时省略，避免标准与紧凑布局拥挤。
- 异常数继续固定显示。
- 该变更不修改侧栏、顶栏或三档窗口布局。

## 8. 日志降噪

- 大批量逐文件“云端未更新，跳过下载”和“内容未变化”日志降为 DEBUG。
- 运行摘要、上传/下载成功、失败、冲突和删除事件继续保留原级别。
- 目的仅是减少大目录同步时的控制台与文件写入，不改变事件持久化和活动统计。

## 9. 验收标准

- 已运行任务在重启后不会早于剩余周期触发；新任务仍会立即首次执行。
- 同一任务上传和下载检查点互不影响。
- 500 个签名未变路径不会调用上传执行器，也不会创建活动运行。
- 未到期墓碑不触发周期上传；同一墓碑重复发现只产生一次 `delete_pending`。
- 纯成功零活动运行在新写入和 schema v4 历史迁移后均不出现在默认活动列表。
- 含删除事实的运行摘要明确显示“删/待删/删失败”。
- 后端全量 pytest、前端测试、ESLint、TypeScript、生产构建和 Windows NSIS 构建通过。
- 不安装、不启动本地构建，不修改用户正在运行的正式数据库；正式体验通过 GitHub Release 自动升级验证。
