# LarkSync v0.8.6 启动后台维护性能热修复

## 1. 问题结论

`v0.8.5` 已把问题页面 GET 与历史回溯分开，但首次升级仍自动从历史头部重放问题事件；日志维护也会从 JSONL 断点逐批向已包含相同数据的 SQLite 回填。两个后台任务同时运行时，页面请求只能等待数据库与事件循环恢复。

正式版只读实测：

- `sync_run_events`：2,514,619 条。
- 问题历史游标一度剩余 36,035 条相关事件。
- `problem_recovery_facts`：已写入 8,032 条时，问题批次仍可阻塞 44–66 秒。
- JSONL 回填断点：6,420,157 / 1,162,403,128 bytes；当前批次连续 `inserted=0, skipped=200`，说明正在重复检查 SQLite 已有事件。
- `/sync/tasks/overview`、`/sync/logs/sync` 实测等待约 44–66 秒；后台繁忙期间 `/health` 也曾超过 20 秒。

## 2. 修复目标

- 启动后立即可用，页面请求不得等待历史回放。
- 实时问题处理只消费应用启动基线之后的新事件。
- `v0.8.5` 已推进的历史游标和全部原始事件继续保留，不删除、不重写。
- 已有 SQLite 事件时，JSONL 重复回填在启动阶段直接对齐到文件尾。
- 正常运行中新写入 JSONL 但 SQLite 写入失败的事件仍可由后续维护恢复。

## 3. 问题游标分层

### 3.1 实时游标

- 新键：`problem_event_cursor_v4`。
- 数据库初始化和中断运行恢复完成后、日志维护与同步调度启动前建立。
- 若 v4 已存在，按原值继续，不能重置。
- 若 v4 不存在，以当前 `sync_run_events` 最大 `(timestamp, id)` 为基线。
- 基线建立后才启动同步调度，因此之后产生的失败和恢复事件不会被跳过。

### 3.2 历史游标

- `problem_event_cursor_v3` 不再作为实时消费位置。
- 首次建立 v4 时，将 v3 当前值复制到 `problem_history_cursor_v3`；若历史键已存在则不覆盖。
- v0.8.6 不自动继续历史回放。
- 历史原始事件、问题、出现记录和恢复事实全部保留；后续可以在明确的空闲维护或手动预演功能中继续。

### 3.3 实时批次

- 单批从 250 降为 20 条。
- 批次满时至少等待 2 秒，追平后每 15 秒检查。
- 问题源异常后等待 30 秒重试。
- 没有新失败事件时，不加载全部问题记录。
- 失败事件只查询本批涉及的指纹；恢复事件继续按解决键精确匹配。

## 4. JSONL 日志回填

### 4.1 启动对齐

在 `SyncLogMaintenanceService.start()` 创建循环前执行一次只读判断：

1. 读取 JSONL 回填状态。
2. 若已完成，不处理。
3. 若未完成且 SQLite 已存在事件，说明当前安装已经具有数据库事件事实；将回填 offset 对齐到当前 JSONL 文件尾并标记 completed。
4. JSONL 文件本身不删除，原始历史仍可审计或以后手动导入。
5. 若 SQLite 为空，则保留原回填逻辑，兼容真正需要从 JSONL 初始化数据库的旧版本。

### 4.2 运行期恢复

- 启动完成后新增事件仍同时写 JSONL 和 SQLite。
- SQLite 正常写入时，维护批次会快速跳过少量重复事件并追平。
- SQLite 写入失败但 JSONL 成功时，维护批次仍会补入缺失事件。
- 本次不删除 JSONL，也不修改保留天数策略。

## 5. 启动顺序

```text
初始化数据库
  -> 恢复中断运行
  -> 建立问题实时游标 v4
  -> 对齐冗余 JSONL 回填断点
  -> 启动日志维护
  -> 启动同步调度
  -> 启动更新调度
  -> 接受页面请求
```

问题实时游标必须早于同步调度建立；否则启动窗口内产生的新失败可能被错误包含进基线而跳过。

## 6. 兼容性

- `v0.8.5 -> v0.8.6`：保留 v3 历史位置，创建 v4 当前基线，立即停止自动历史追赶。
- `v0.8.4 -> v0.8.6`：已有统一问题直接可读；v4 从现有事件尾开始处理新事实。
- 更老版本且 SQLite 事件表为空：JSONL 回填继续执行，不会直接跳到文件尾。
- 全新安装：v4 从空位置开始，之后每个新事件正常处理。
- 不包含任何用户路径、任务 ID 或正式数据库专用值。

## 7. 验收标准

- 升级后问题 v4 游标立即等于启动前最新事件；v3 和历史游标值保持一致。
- 同步调度启动后追加的新失败事件能进入问题中心。
- 已有 SQLite 事件且 JSONL 回填未完成时，启动对齐后 offset 等于文件大小，状态 completed。
- SQLite 为空时仍按原批次导入 JSONL。
- 后端全量 pytest、前端测试、Lint、类型检查和生产构建通过。
- 使用正式数据库只读证据确认 v0.8.5 阻塞原因；不直接修改正式数据库验证热修复。
- Windows 本地只构建安装包，不安装、不启动；正式体验由 GitHub Release 自动升级验证。
