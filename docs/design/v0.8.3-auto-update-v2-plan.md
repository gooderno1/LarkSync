# 自动更新 v2 升级方案

版本：v0.8.3 方案稿 v1  
日期：2026-07-22  
状态：已完成现状审计与升级设计，尚未进入代码实现

## 1. 结论

当前自动更新不是“完全没做”，而是已经分别实现了检查、下载、校验、安装请求、托盘接管、静默安装和重启回执；问题在于这些能力没有被同一个持久化任务状态机串起来。

页面上的五步流程目前主要根据三个快照临时推导：

1. `status.json` 中是否存在 `download_path`。
2. `install-request.json` 是否存在。
3. `install-handoff.json` 当前最后一个阶段是什么。

这种推导无法表示下载进度、网络重试、失败原因、断点恢复和历史阶段，也无法确认三个文件是否属于同一次更新。因此页面虽然有流程外观，但还不是一个真实、连续、可恢复的更新流程。

建议在 v0.8.3 引入“更新操作 UpdateOperation”作为唯一事实来源，先修复下载可靠性和错误可见性，再统一安装交接与前端流程展示。

## 2. 本次实机证据

### 2.1 下载失败

v0.8.1 下载 v0.8.2 时出现两次真实中断：

- 第一次接收 `21,036,352 / 57,021,601` 字节后，远端提前关闭响应体。
- 第二次接收 `4,656,256 / 57,021,601` 字节后，远端提前关闭响应体。
- 异常类型：`httpx.RemoteProtocolError`。
- 路由只捕获 `RuntimeError`，未捕获该网络异常，最终成为 HTTP 500。
- `update.log` 没有记录这两次失败，异常只出现在后端 stderr。
- 第三次重试完整下载并通过 SHA256 校验，说明 Release 文件本身正常，失败来自传输链路中断。

### 2.2 安装完成后的流程错位

v0.8.2 已于 2026-07-22 成功安装并自动重启。当前真实状态为：

- `current_version = v0.8.2`。
- `latest_version = v0.8.2`。
- `update_available = false`。
- `download_path = null`。
- `install_request = null`。
- `install_handoff.stage = restart_succeeded`。

现有前端会由此同时推导出：

- “校验通过：等待下载”。
- “托盘接管：等待确认”。
- “helper 启动：已接管”。
- “静默安装：已完成”。
- “自动重启：已确认”。

这是互相矛盾的展示。根因是完成后的 handoff 被保留，而下载路径和安装请求按各自规则清空；前端没有以 `request_id + target_version` 识别同一次操作，也没有完成态归档和启动后对账。

## 3. 当前实现盘点

### 3.1 已实现能力

| 能力 | 当前实现 | 当前结果 |
| --- | --- | --- |
| Release 检查 | GitHub API；403/429 时回退公开 Release 跳转 | 可用 |
| 平台包选择 | Windows EXE；macOS 按架构选择 DMG | 可用 |
| 完整性校验 | GitHub digest、`.sha256`、Release 正文三级来源 | 可用 |
| 手动下载 | 后端同步流式下载 | 能完成，但不可靠 |
| 自动下载 | `UpdateScheduler` 每分钟检查，开启开关后下载新版本 | 能完成；“自动更新”实际只是自动下载 |
| 安装确认 | 前端确认框后写入安装请求 | 可用 |
| 托盘接管 | 托盘轮询安装请求并启动 helper | 可用 |
| Windows 静默安装 | PowerShell bootstrap/worker + NSIS `/S` | 已有较完整 smoke |
| 重启确认 | helper 多次确认重启进程并写 handoff | 可用 |
| 页面流程 | 五步卡片展示下载、接管、安装、重启 | 仅快照推导，未形成真实流程 |

### 3.2 关键缺口

#### P0：下载可靠性

- 直接写最终安装包文件，没有 `.part` 临时文件和原子重命名。
- 没有按字节记录进度、速度、剩余时间和尝试次数。
- 手动下载没有自动重试。
- 没有 HTTP Range 断点续传；网络中断后只能整包重下。
- 网络异常未统一转换为业务错误，可能直接返回 HTTP 500。
- 下载失败未写 `status.json` 和 `update.log`，用户和开发者都无法从更新页定位。
- 下载接口是长时间同步请求，前端只能显示“下载中”，无法查询后台进度。

#### P0：流程状态不一致

- 下载、安装请求和 helper 回执分散在三个文件中，没有统一操作 ID。
- `install_request` 被托盘消费后会删除，导致页面无法知道托盘接管已完成。
- `install_handoff` 完成后保留，可能污染下一次更新或已升级页面。
- 前端没有校验 `install_handoff.request_id` 是否与当前安装请求一致。
- 安装成功后的新版本启动没有执行“当前版本是否等于目标版本”的最终对账。
- `useUpdate` 没有活动阶段轮询；失去页面请求后无法连续观察交接状态。

#### P1：自动更新语义

- 配置名为“自动更新”，实际行为是“自动检查并下载”，不会自动安装。
- 没有“仅提醒 / 自动下载 / 空闲时自动安装”的明确策略。
- 没有任务运行检查、安装时间窗、倒计时取消或延期机制。
- 页面没有说明 Windows 可能出现 UAC、应用会退出以及未保存工作的风险。

#### P1：可观测性和维护

- `last_error` 只可靠承载检查错误，不能表示下载和安装错误。
- 单个 handoff 文件只保留最后阶段，没有阶段历史。
- 没有结构化错误码、是否可重试、下次重试时间和诊断建议。
- 历史安装包长期保留，没有按版本数或空间阈值清理。

## 4. 目标状态机

### 4.1 单一更新操作

新增持久化对象 `UpdateOperation`，每一次发现并处理目标版本时创建一个稳定 `operation_id`。检查结果仍可缓存，但下载、校验、安装和重启都必须写入同一个操作。

建议阶段：

1. `idle`：没有活动更新。
2. `checking`：正在获取 Release。
3. `available`：发现可用版本，等待策略决策。
4. `download_queued`：已进入后台下载队列。
5. `downloading`：正在下载，持续更新字节进度。
6. `download_retry_wait`：传输中断，等待自动重试。
7. `verifying`：校验文件大小和 SHA256。
8. `ready_to_install`：安装包已原子落盘并可安装。
9. `install_queued`：用户确认或策略允许，等待托盘接管。
10. `helper_starting`：托盘已消费请求，正在启动 helper。
11. `installer_started`：安装器进程已启动。
12. `installing`：安装器仍在运行。
13. `restart_pending`：安装完成，准备拉起新版本。
14. `restart_succeeded`：新进程已确认。
15. `completed`：新版本启动后对账通过，操作归档。
16. `failed`：不可继续的终态失败。
17. `cancelled`：用户在允许阶段取消。

`download_retry_wait`、`failed` 必须同时保存：错误码、用户可读消息、技术详情、是否可重试、已尝试次数和下一次重试时间。

### 4.2 状态推进原则

- 所有状态写入使用临时文件 + 原子替换，避免进程退出时 JSON 半写。
- 状态只允许按定义的边推进；回退必须生成新的重试记录，不能静默修改旧阶段。
- 每个操作绑定 `target_version`、`asset_name`、`expected_size`、`expected_sha256`。
- 安装请求和 handoff 必须携带同一个 `operation_id`、`request_id` 和 `target_version`。
- 当前版本启动后，若 `current_version == target_version`，把操作最终标记为 `completed`。
- 若 handoff 属于旧操作，状态接口只把它放入历史，不参与当前流程推导。

## 6. 下载器升级设计

### 6.1 后台任务化

- `POST /system/update/download` 改为启动后台下载并返回 HTTP 202 及 `operation_id`。
- 下载由单实例 `UpdateDownloadManager` 管理，同一目标版本只允许一个活动任务。
- `GET /system/update/status` 返回当前操作进度；活动阶段前端每 1 秒轮询，静态阶段恢复 30 秒缓存。
- 关闭页面或隐藏到托盘不终止下载；后端退出时保存断点信息。

### 6.2 临时文件与断点续传

- 下载目标使用 `<asset>.part`，校验通过后原子重命名为正式文件。
- 若 `.part` 已存在，发送 `Range: bytes=<size>-`。
- 服务器返回合法 `206 + Content-Range` 时续传。
- 服务器忽略 Range 返回 200 时，从头覆盖 `.part`，不能把完整响应追加到旧文件。
- 校验 `Content-Length` / `Content-Range` 与 Release 预期大小；大小不符不得进入 SHA 校验成功态。
- SHA256 通过后才写 `installer_path`；失败时保留诊断并删除不可信正式文件。

### 6.3 重试策略

- 默认最多 4 次：立即下载 + 3 次自动重试。
- 退避建议：2 秒、5 秒、15 秒，并加入小幅随机抖动。
- 可重试：连接失败、读取超时、`RemoteProtocolError`、HTTP 408/425/429/500/502/503/504。
- 不可重试：HTTP 401/403（除非重新检查后资产 URL 改变）、404、校验信息缺失、重复 SHA 不匹配。
- 429 优先遵守 `Retry-After`。
- 每次失败写 `update.log` 和结构化 operation 事件；禁止只进入 stderr。

### 6.4 错误接口

- API 层必须统一捕获 `httpx.HTTPError`、`OSError` 和更新业务异常。
- 后台任务失败时接口仍返回可查询状态，不再把传输异常直接变成无正文的 500。
- 错误码示例：`UPDATE_NETWORK_INTERRUPTED`、`UPDATE_HTTP_ERROR`、`UPDATE_SIZE_MISMATCH`、`UPDATE_SHA256_MISMATCH`、`UPDATE_DISK_WRITE_FAILED`、`UPDATE_CANCELLED`。
- 前端展示用户可读说明；技术详情折叠显示并支持复制。

## 7. 安装交接升级设计

### 7.1 统一关联

- `queue_install_request()` 增加 `operation_id` 和 `target_version`。
- helper 的每次 handoff 都写相同关联字段。
- 托盘消费请求时不直接让“是否存在请求文件”承担流程状态；先把操作推进到 `helper_starting`，再删除队列文件。
- handoff 仍作为跨进程通信文件，但由后端合并进 `UpdateOperation`，前端不再直接把它推导成五步状态。

### 7.2 新版本启动对账

新版本后端启动时执行一次快速本地对账：

1. 读取最后一个未完成操作。
2. 比较当前版本与 `target_version`。
3. 相等则标记 `completed`，记录“版本对账通过”。
4. 不相等且安装器已成功退出，则标记 `failed`，错误码为 `UPDATE_VERSION_NOT_APPLIED`。
5. 将旧 handoff 归档，不再污染当前更新检查。

### 7.3 安全边界

- 默认策略仍为“自动下载，手动确认安装”。
- 安装前检查同步任务是否运行；运行中则进入 `waiting_for_idle` 或提示稍后安装。
- 安装确认框明确说明应用会退出、后台同步会暂停、Windows 可能弹出 UAC。
- 任何下载或校验失败都不能生成安装请求。

## 8. 前端升级设计

### 8.1 流程区

把现有静态五步卡改为由 `UpdateOperation.stage` 驱动的真实流程：

- 检查更新。
- 下载更新。
- 完整性校验。
- 安装交接。
- 安装与重启。
- 新版本确认。

只显示当前操作的阶段。已经升级完成时展示“v0.8.2 已安装并完成重启确认”的完成摘要，不再同时显示“等待下载”。

### 8.2 下载体验

- 显示 `已下载 / 总大小`、百分比、速度、预计剩余时间。
- 自动重试时显示“连接中断，5 秒后进行第 2/4 次尝试”。
- 提供“立即重试”和“取消下载”；支持断点时提示“将从 20.1 MB 继续”。
- 失败卡展示错误码、用户可读原因、复制诊断和打开更新日志。

### 8.3 安装体验

- 安装开始后每 1 秒刷新状态，直到应用退出。
- 新版本重启后恢复同一操作并显示最终完成结果。
- `restart_failed` 解释为“安装可能已完成，但未确认新程序持续运行”，并提供“重新启动 LarkSync”和“打开安装包目录”。
- 历史更新结果单独收起，不与当前流程共用状态。

### 8.4 自动更新策略文案

把单一复选框改为明确策略：

1. `仅提醒`：检查到新版本后通知，不下载。
2. `自动下载，手动安装`：推荐默认值。
3. `空闲时自动安装`：后续增强；仅在无同步任务运行、处于维护时段且用户未延期时执行。

v0.8.3 第一阶段建议先实现前两项，避免一次版本同时扩大下载与自动安装风险。

## 9. API 调整建议

| 方法 | 路径 | 作用 | 返回 |
| --- | --- | --- | --- |
| GET | `/system/update/status` | 当前 Release + 当前操作 +最近完成结果 | 200 |
| POST | `/system/update/check` | 立即检查，并创建或更新可用版本操作 | 200 |
| POST | `/system/update/download` | 启动或恢复后台下载 | 202 |
| POST | `/system/update/download/retry` | 立即重试可重试失败 | 202 |
| POST | `/system/update/download/cancel` | 取消下载，保留或清理 `.part` 按策略执行 | 200 |
| POST | `/system/update/install` | 校验 ready 状态并排队安装 | 202 |
| GET | `/system/update/history` | 最近更新操作与阶段历史 | 200 |

保留原有字段一个兼容周期，但新前端只消费 `operation`；旧 `download_path` 由 operation 的 ready 状态派生。

## 10. 测试方案

### 10.1 后端单元测试

- 连接在 4.4 MB 和 20.1 MB 中断时进入重试，不返回原始 500。
- `.part` 存在且服务端支持 Range 时从正确字节续传。
- 服务端忽略 Range 时安全重下，不追加损坏文件。
- 429 遵守 `Retry-After`；503 按退避重试。
- 下载大小不符不能进入 `verifying` 成功态。
- SHA 不符删除正式目标并保留结构化错误。
- 四次均失败时进入可解释的 `failed`，update.log 有四次尝试记录。
- 已有大小和 SHA 均匹配的正式包直接进入 `ready_to_install`。

### 10.2 状态机测试

- 非法逆向状态转换被拒绝。
- 安装请求消费后，操作仍保持 `helper_starting`，不回到“等待确认”。
- handoff 的 request_id 不匹配时不污染当前操作。
- 新版本启动且版本匹配时进入 `completed`。
- 当前版本已经等于最新版本时，旧 handoff 只出现在历史。

### 10.3 前端测试

- 每个活动阶段能展示准确文案、进度和可用动作。
- `restart_succeeded + current==target` 只显示完成态。
- 下载中断显示自动重试倒计时，不显示泛化“请求失败 500”。
- 活动阶段启用 1 秒轮询，完成或失败后停止高频轮询。
- 自动更新策略文案与实际行为一致。

### 10.4 安装包体验测试

- Windows：正常网络、中途断网、代理重启、下载后应用重启、UAC 取消、安装器失败、新版本未拉起。
- macOS：双架构 DMG 选择、下载恢复、手动安装引导和新版本对账。
- 正式版旁测试必须使用独立更新目录或专用测试 Profile，不覆盖正在使用的生产安装。

## 11. 验收标准

- 57 MB 安装包在两次连接中断后无需用户重新从零下载，最终 SHA256 正确。
- 下载阶段页面每 1 秒内更新进度，字节数与落盘 `.part` 大小一致。
- 所有失败均有结构化错误码、用户可读原因和 update.log 记录。
- 更新流程任意时刻只出现一个自洽阶段，不再同时展示“等待下载”和“自动重启成功”。
- 安装完成并重启后 10 秒内完成版本对账并归档操作。
- “自动更新”设置文案与实际策略一致，默认不会未经确认自动退出应用。
- 原有 Windows helper、NSIS 静默安装和 macOS 架构选择回归测试继续通过。

## 12. 实施顺序与预估

1. **状态模型与失败测试（1 天）**：建立 `UpdateOperation`、阶段转换、原子持久化和真实中断测试。
2. **后台下载器（1.5～2 天）**：`.part`、Range、自动重试、进度与结构化错误。
3. **安装交接统一（1～1.5 天）**：operation/request/handoff 关联、启动对账和历史归档。
4. **前端流程实装（1～1.5 天）**：真实阶段、进度、重试/取消、完成摘要和策略文案。
5. **安装包与网络回归（1 天）**：Windows 真实安装 smoke、代理中断和 macOS 定向回归。

预计总计 5.5～7 个开发日。建议作为 v0.8.3 的独立主任务，不与活动管理 / 问题中心页面实现放在同一个发布提交中；两项可以同版本开发，但应拆分提交、测试和回滚边界。

## 13. 本方案边界

- 本文只定义 LarkSync 内部状态和接口，不新增或猜测飞书 API 字段。
