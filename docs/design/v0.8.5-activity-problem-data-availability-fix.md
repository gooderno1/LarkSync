# LarkSync v0.8.5 活动与问题数据可用性修复设计

## 1. 修复目标

- 标准模式只保留左侧任务列表和顶部运行选择，不重复显示任务选择器。
- 活动页面必须展示已持久化的真实运行与事件；接口失败时显示错误和重试入口，不能伪装成空列表。
- 旧版无动作检查记录归档保留，默认不显示、不计入活动统计。
- 问题中心首屏只读取已经持久化的问题，不在 GET 请求中同步执行历史回填。
- 历史问题与后续成功事件通过后台增量游标关联，避免扫描全部事件表并阻塞 SQLite。

## 2. 已确认的数据事实

- 正式版数据库包含 18,173 条真实活动运行和 42,991 条旧版无动作检查。
- 正式版数据库包含 3,084 条未解决问题、11 条已解决问题和 39,782 条问题证据。
- 活动与问题页面显示为空不是数据缺失，而是问题历史回填长时间占用 SQLite，导致活动总览和问题查询超时。
- 前端使用全零占位数据，使加载中或超时状态被误显示为“0 条”。

## 3. 活动管理布局

### 3.1 标准模式

- 左栏：唯一的任务选择入口，包含搜索和任务列表。
- 主栏顶部：仅显示当前运行选择器，不再重复任务选择器。
- 主栏正文：事件筛选、搜索、事件列表、分页和事件详情。
- 默认时间范围：全部保留记录，确保任务切换后能立即看到最近一次真实活动。

### 3.2 紧凑模式

- 顶部保留任务选择器和运行选择器。
- 事件详情使用列表到详情的单页切换。

### 3.3 宽屏模式

- 三栏依次为任务列表、运行列表、事件列表。
- 不额外显示运行下拉框。

### 3.4 错误与空状态

- 总览、运行或事件请求失败时显示明确错误说明和重试按钮。
- 只有请求成功且结果确实为空时才显示空状态。
- 加载中不使用零值占位统计冒充真实结果。

## 4. 历史运行兼容

- `run_kind=legacy_check` 且 `has_activity=false` 的旧记录继续保存在 `sync_runs`。
- 默认运行查询保持 `has_activity=true`，因此旧版无动作检查不进入活动管理。
- 有上传、下载、删除、冲突、待删除、失败或取消状态的旧运行仍视为真实活动。
- 本版本不删除、不改写旧记录主键和时间字段，保证升级回滚和数据库审计兼容。
- 如后续需要查看归档检查，应增加独立的“历史检查”入口，不能重新混入默认活动列表。

## 5. 问题中心数据链路

### 5.1 前台查询

- `/problems` 与 `/problems/summary` 默认 `refresh=false`。
- 页面轮询只读取 `problems`、`problem_occurrences` 和 `problem_actions`。
- 首次请求不注入全零占位数据；加载、失败、真实为空三种状态分开显示。

### 5.2 后台增量处理

- 使用 `sync_meta.problem_event_cursor_v3` 保存最后处理的事件时间戳和事件 ID。
- 每批只按 `sync_run_events.timestamp, id` 顺序读取固定数量的失败或恢复事件。
- 失败事件写入问题及证据；同任务、同对象、同操作族的后续成功事件写入恢复事实并结案。
- 游标和问题变更在同一事务提交，进程中断后允许安全重试。
- 启动后台任务先短暂让出资源，再按小批次处理；批次未满时降低轮询频率。
- 不再为每次页面查询构造数千路径的 `LOWER(REPLACE(path)) IN (...)` 全表匹配。

### 5.3 兼容与边界

- 已有问题、证据和人工处理记录全部保留。
- 首次升级从历史事件头开始分批补齐恢复事实，期间问题列表立即可读。
- `delete_pending` 仍是工作流状态，不创建问题；旧版误建记录按既有规则标为已解决。
- 无变化检查不能作为问题已解决的证据。

## 6. 验证标准

- 标准模式 HTML 中只有一个任务选择区域，且主栏只出现运行选择器。
- 旧版无动作检查默认查询不返回，数据库原始记录数量不减少。
- 有 250 万事件的数据库上，活动总览和问题列表不会等待历史问题回填完成。
- 问题 GET 接口不调用 `refresh_sources`。
- 增量处理在失败后成功、不同对象成功、重复运行和进程重启场景下均通过测试。
- 正式包完成后验证安装包、更新元数据、SHA256 和 GitHub Release 资产。
