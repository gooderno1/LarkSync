# Release Standard

LarkSync 的正式版 Release 说明从本文件起执行以下标准。目标只有一个：让用户在 GitHub Release 页面上，能直接看清楚“这次到底改了什么、影响了什么、该注意什么”。

## 适用范围

- 所有稳定版发布：`vX.Y.Z`
- 不适用于开发版：`vX.Y.Z-dev.N`

## Release 说明必须包含的结构

1. 版本标题  
   例如：`# LarkSync v0.6.2`

2. 发布元信息  
   必须写清楚：
   - 发布日期
   - 变更区间（上一个稳定版 -> 当前稳定版）

3. `升级重点`  
   这一段必须先出现，且必须是用户可读的结果导向描述。  
   不能只写 `fix(sync)`、`feat(ui)` 这类提交摘要。

4. `详细变更`  
   必须按本次稳定版纳入的各个 `dev` 版本展开，逐条列出每个开发版本的实际结果。

5. `安装包校验`  
   如存在安装包资产，必须附带 sha256。

## 内容要求

- 必须说明“行为变化”而不是只写“做了某个提交”。
- 必须明确影响对象，例如：
  - 自动更新
  - 冲突管理
  - 日志中心
  - 删除联动
  - Markdown 上下行同步
- 必须避免模糊表述，例如：
  - “优化了一些问题”
  - “修复若干 bug”
  - “提升稳定性”
- 如果涉及用户可见的限制或边界，也必须写明，例如：
  - 安装在 `Program Files` 时仍可能弹出 UAC
  - 只在 Windows 生效
  - 仅针对 Markdown/Docx 场景

## 数据来源标准

- Release 说明主数据源：`DEVELOPMENT_LOG.md`
- 变更区间判定：`CHANGELOG.md`
- 安装包校验：构建产物实际 sha256

## 执行要求

- 发布稳定版前，必须先确认 `DEVELOPMENT_LOG.md` 中对应 `dev` 版本的“结果”已补全。
- 若 `DEVELOPMENT_LOG.md` 缺少对应记录，不允许用模糊摘要直接发布，必须先补文档。
- `scripts/release_notes.py` 生成的说明即为默认发布说明模板；后续发布不得手工降级为更模糊的版本。
